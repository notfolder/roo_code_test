# リポジトリ概要

Roo Codeエージェントを使って「要件定義 → 詳細設計 → コード生成」というウォーターフォール型AI開発ワークフローを検証するためのプロンプト実験リポジトリ。人間はプロジェクトマネージャーとして振る舞い、AI エージェントがソフトウェアエンジニアリングを担う。

## ワークフロー

```
ユーザー（PM） → 要件定義エージェント → 詳細設計エージェント → コーディングエージェント
```

1. **要件定義** (`agents/REQUREMENTS_DEFINITION_AGENT.md`): ユーザーへのヒアリングで完全な要件定義書を生成。出力は `doc/` ディレクトリ配下に Markdown で保存。
2. **詳細設計** (`agents/DESIGN_SPEC_AGENT.md`): 要件定義書から矛盾のない詳細設計書を生成。コード例は書かず日本語で記述、図は Mermaid。
3. **コード生成** (`agents/CODING_AGENT.md`): 設計書からモックなし・完全な動作コードを生成。実装後に必ず自己レビューを実施。

## 生成アプリの技術スタック

エージェントが生成するアプリのデフォルト構成：

- **言語**: Python（特段の理由がない限り）
- **GUI（シンプル）**: Streamlit
- **GUI（複雑な画面遷移）**: Vue.js + Vuetify（フロントエンド）+ FastAPI（バックエンド）
  - フロントエンド: マルチステージビルド（npm build → nginx）
  - バックエンド: nginx でリバースプロキシ、APIエンドポイントは `/api/` 以下
- **インフラ**: Docker Compose（必須）
- **DB初期化**: システム起動時に自動でスキーマ構築・初期データ投入

## コーディング規約（生成コードへの適用）

- コメントは **必ず日本語** で記述する
- モックアップ・省略コードは **絶対禁止**。動作する完全なコードを出力する
- コード作成・修正後は **必ず自己レビュー** を実施し、関連コードへの影響・冗長性を確認して報告する
- 同一処理を複数箇所に再実装することを禁止（DRY原則の徹底）
- 2回以上使う処理は必ず共通モジュールに切り出す

## ドキュメント規約（要件定義書・詳細設計書）

- Markdown 形式、図は Mermaid
- コード例・将来拡張案・実装スケジュールは **絶対に記述しない**
- 要件定義書にはクラス名・関数名・ファイル名などの設計情報を含めない
- 詳細設計書には処理内容を日本語で具体的に記述（コードは書かない）
- 作成・修正後は必ず矛盾・冗長性のレビューを実施して結果を報告する

## 回答・ドキュメントの言語

ユーザーへの回答、設計書、コードコメントはすべて **日本語** で記述する。

## 開発対象アプリ：社内備品管理・貸出管理システム

要件定義書: [doc/requirements.md](doc/requirements.md)

### アプリ概要

社内備品（PC・プロジェクター等）の貸出状況をリアルタイムで一元管理するWebアプリ。Excelによる記載ミスや貸出先不明問題を解消する。

### ユーザーと権限

| 役割 | できること |
| ------- | ----------- |
| 管理担当者 | 備品の登録・編集・削除、貸出・返却操作、貸出履歴確認、アカウント作成・削除 |
| 一般社員 | 貸出可能な備品一覧の参照のみ |

### 主要エンティティ

| エンティティ | 主な項目 |
| ----------- | -------- |
| 備品 | 管理番号（一意）、備品名、備考、貸出状態（貸出可能／貸出中） |
| アカウント | アカウント名（一意）、パスワード（ハッシュ化）、役割 |
| 貸出記録 | 貸出日時、返却日時、備品、借りた社員のアカウント名 |

### 備品の状態遷移

```text
備品登録 → 貸出可能 ⇄ 貸出中（管理担当者が操作）→ 備品削除
```

### 画面構成

- ログイン画面（全ユーザー）
- 備品一覧画面（全ユーザー：貸出可能備品の参照）
- 貸出・返却管理画面（管理担当者のみ）
- 備品管理画面（管理担当者のみ）
- アカウント管理画面（管理担当者のみ）
