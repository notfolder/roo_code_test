# 出力形式
- markdown形式
- 図はmermaidで記述
- コード・設計要素（クラス/関数等）は記述しない

# 要件定義内容
- 今回の変更要望を実現するために必要な要件を整理します
- 各要件は「業務課題を解決するために必要な最小単位」に限定すること
- 既存の要件と同じ目的を持つ要件は変更要件に加えないこと
- 将来拡張・一般論・ベストプラクティスは記述しないこと

この上で、以下をすべて今回の要求内容としてまとめること：

## 1. 目的・前提
- 今回の変更の目的を必ず確認すること
- 今回の変更で解決すべき課題を必ず確認すること
- 用語集に追加する用語（不明用語は必ずユーザーに確認）
- CUIであれば、GUIに変更するかどうか

## 2. 業務

今回の変更で対象業務に追加・変更・削除があるか？を必ず確認する。
追加変更削除のある業務について、以下をすべて含めること：

- 業務のフロー（mermaid）
- 業務の範囲・担当者
- 業務の業務課題・KPI（リードタイム、エラー率等）
- 今回解決すべき課題と対応方針
- 今回の変更による見込み経営効果を記述(Hard Saving装置削減,Soft Saving人件費削減,Net working capital,Cost Avoidance,Total Cost of Ownership Savings,etc..)
- 今回解決すべき課題に対応する機能について、追加か変更、削除のどれかを明示すること
- 今回の変更で追加・変更される機能は**必ず**業務課題に紐づけること、紐づかない機能はユーザーに確認して削除することを明記すること

## 3. 機能要件
今回の変更で追加・変更される機能について、以下をすべて含めること：
削除する機能については明示すること。

- 機能一覧（業務課題から導出、必要な機能が全てあることを確認）
- 入力データ（人手/外部）
- 出力データ
- 外部連携
- GUIなら全画面の遷移、CUIなら全引数仕様
- 機能についてのユーザーの利用フロー（mermaid）
- 業務フローとの対応関係

## 4. データ

今回の変更で追加・変更されるデータについて、以下をすべて含めること：
削除するデータについては明示すること。

- 業務エンティティ一覧（業務で扱うデータ）
- 内部データ/外部データ
- データ保持期間
- 外部DB接続先,接続方法の一覧

## 5. 非機能要件
今回の変更で変更が必要な非機能要件があるか確認し、ある場合は以下をすべて含めること：
- 性能（応答時間）
- 利用人数（同時接続）
- セキュリティ

## 6. テスト用利用シナリオ
- 今回の変更で追加・変更される機能に対して、ユーザーの利用フローについて、フローの網羅性を100%満たすシナリオを作成し、表形式で記述すること
- GUIの場合、全画面の遷移を網羅し、操作方法を具体的に記述すること
- 各シナリオに対して、テストの目的、前提条件、テスト手順、期待される結果を表にすること

# 要件網羅性チェック（必須）

今回の変更で追加・変更される要件が以下を満たさない場合は不足として必ず指摘する：

- 業務エンティティを完全に列挙する
- 各エンティティに対して以下を必ず定義する：
  CRUD / 一覧 / 詳細 / 検索 / 状態（ステータス）
- マスタ該当エンティティには管理機能を必ず含める
- 機能カテゴリを網羅する：
  業務機能 / マスタ管理 / 共通（認証・認可・ログ）/ 運用 / 外部連携
- 各エンティティの状態遷移を定義する
- 機能と画面・ユーザーフローの対応を検証する
- 作成した要件から「削除可能な要件」を列挙すること
- 削除しても業務が成立する要件は削除すること
- MVPに必要最低限の要件のみであることを確認すること

変更要件仕様書の作成は既存要件仕様と合わせて、以下のルールに従ってください。

- ユーザーに質問する際は、具体的かつ明確な質問を一つづつ行い、必要な情報を引き出すこと
- 要件仕様書を書く際にはコード例は絶対に書かず、日本語で内容を具体的にに記述すること
- 要件仕様書を書く際には将来の拡張案は絶対に記述しないこと
- 要件仕様書を書く際には実装スケジュール/実装フェーズは絶対に記述しないこと
- 要件仕様書はmarkdown形式で記述し、複雑な説明にはmermaid形式の図を描くようにすること
- 要件仕様書にはクラス名や関数名、ファイル名などの設計に関わる情報はは絶対に記述しないでください。
- 要件仕様書を作成,修正したら**絶対に、必ず**内容をレビューして内容に**矛盾はないか**、**冗長ではないか**などを確認、修正して**必ず**レビュー結果を報告してください

- 質問は一度に一つ、具体的に行う
- 要件作成 → 抜け漏れチェック → 修正 を繰り返す
- 矛盾・不足は必ず指摘する
- 完全性確認後にレビューし、問題がなければ完了として.history/[YYYYMMDD]-[タスク名]/change_requirements.mdファイルを保存する
