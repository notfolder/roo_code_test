要件定義とかファイルにしないことがあるので、定義させる
出来るだけ単純にしたいが、マスター系が作成されず結果使えないことが多い。
→画面遷移図とシステム充足をチェックさせる方法を検討する必要がある

以下を必ず満たして機能を生成せよ：

- エンティティを列挙し、各エンティティにCRUD＋一覧＋詳細＋検索＋状態管理を定義
- マスタ該当エンティティには必ず管理画面を付与
- 以下カテゴリをすべて含める：
  業務 / マスタ / 共通（認証・権限・ログ）/ 運用 / 外部連携 / 非機能
- 各エンティティに状態遷移を定義
- 各機能に一覧・詳細・編集画面を定義
- 標準的な同種システムと比較し不足機能を列挙
- 最後に抜け漏れ候補を必ず出力
- 各エンティティに状態遷移を定義すること
- 機能と画面の対応関係を双方向で検証すること


# 網羅性チェック（必須）

- 同種システムの標準機能と比較し不足機能を列挙すること
- データ整合性（制約・トランザクション・排他制御）を定義すること
- 全機能に対応するテストが存在するか検証すること

- プロジェクトのdocディレクトリ配下に適切なファイル名でファイルとして出力すること

# 要件定義内容
以下をすべて含めること：

## 1. 目的・前提
- システム目的
- 用語集（不明用語は必ずユーザーに確認）
- GUIかCUIで良いのか

## 2. 業務
- 対象業務一覧
- 業務フロー（mermaid）
- 業務の範囲・担当者
- 業務課題・KPI（リードタイム、エラー率等）
- 解決すべき課題と対応方針
- システム化をした際の見込み経営効果を記述(Hard Saving装置削減,Soft Saving人件費削減,Net working capital,Cost Avoidance,Total Cost of Ownership Savings,etc..)

## 3. 機能要件
- 機能一覧（業務課題から導出、必要な機能が全てあることを確認）
- 入力データ（人手/外部）
- 出力データ
- 外部連携
- GUIなら全画面、CUIなら全引数仕様
- GUIなら全画面の遷移
- 全機能についてのユーザーの利用フロー（mermaid）
- 業務フローとの対応関係

## 4. データ
- 業務エンティティ一覧（業務で扱うデータ）
- 内部データ/外部データ
- データ保持期間
- 外部DB接続先,接続方法の一覧

## 5. 非機能要件
- 性能（応答時間）
- 利用人数（同時接続）
- セキュリティ

# 網羅性チェック（必須）

  CRUD / 一覧 / 詳細 / 検索 / 状態（ステータス）

  業務機能 / マスタ管理 / 共通（認証・認可・ログ）/ 運用 / 外部連携
- 各エンティティの状態遷移を定義する
- 機能と画面・ユーザーフローの対応を検証する
- 同種システムの標準機能と比較し不足機能を列挙する

ドキュメント作成は以下のルールに従ってください。

- ユーザーに質問する際は、具体的かつ明確な質問を一つづつ行い、必要な情報を引き出すこと
- 要件仕様書を書く際にはコード例は絶対に書かず、日本語で内容を具体的にに記述すること
- 要件仕様書を書く際には将来の拡張案は絶対に記述しないこと

- 要件仕様書にはクラス名や関数名、ファイル名などの設計に関わる情報はは絶対に記述しないでください。
- 要件仕様書を作成,修正したら**絶対に、必ず**内容をレビューして内容に**矛盾はないか**、**冗長ではないか**などを確認、修正して**必ず**レビュー結果を報告してください

# 設計内容
以下をすべて含めること：
- 言語・フレームワーク
  - 言語は特段の理由がなければpythonを選んで
  - webのGUIが必要な場合は、基本的にははStreamlitを選んで。複雑な画面遷移、部品が必要な場合にはvue(vuetify)+バックエンドはFastAPIで実装して
- システム構成（コンポーネント・データフロー・ネットワーク）
  - 要件定義書を基にシステムを構成するコンポーネントの一覧を表にする
  - システム全体の構成を説明するための図をmermaid形式で記述
  - 各コンポーネントの役割と機能を説明
  - コンポーネント間のインターフェースとデータフローを説明
  - ネットワーク構成図をmermaid形式で記述
- データベース設計
  - 要件定義書を基に必要なデータベースのテーブル設計を行う
  - 各テーブルのカラム、データ型、制約条件を表にする
  - テーブル間のリレーションシップを説明するための図をmermaid形式で記述
- 外部設計
  - ユーザーインターフェースの設計
    - GUIの場合、必要な画面のレイアウトや要素を説明するための図をmermaid形式で記述
      - 画面一覧:各画面の要素と機能を説明
      - 画面遷移図:画面間の遷移を説明するための図をmermaid形式で記述
      - AAによる画面のモックアップ:各画面のモックアップをAAで表現
    - CUIの場合、必要なコマンドや引数、設定の一覧を表にする
  - 外部システムとの連携設計
    - 連携する外部システムの一覧を表にする
    - 各外部システムとのインターフェースとデータフローを説明するための図をmermaid形式で記述
  - 外部データベースとの連携設計
    - 連携する外部データベースの一覧を表にする
    - 各外部データベースとのインターフェースとデータフローを説明するための図をmermaid形式で記述
- 内部設計（処理フロー・バッチ）
    - システム内部の処理フローを説明するための図をmermaid形式で記述
    - 各処理の役割と機能を説明
    - 必要であればバッチ処理の設計
      - バッチ処理のフローを説明するための図をmermaid形式で記述
      - 各バッチ処理の役割と機能を説明
- 全クラスの設計(共通部分があるなら継承を使うなど)
  - 全くラスの一覧表
  - 各クラスの役割と機能,主な属性とメソッドを説明
  - クラス間の関係性を説明するためのクラス図をmermaid形式で記述
- オブジェクト指向設計におけるメッセージの整理
  - システム内でやり取りされるメッセージの一覧を表にする
  - 各メッセージの内容と役割を説明
  - メッセージのフローを説明するための図をmermaid形式で記述
- エラーハンドリング
  - システム内で発生する可能性のあるエラーの一覧を表にする
- セキュリティ設計
- ソースコード構成
  - ソースコードのディレクトリ設計を説明するためAAで表現: ディレクトリ構成のみ
  - 各ディレクトリに格納されるファイル名とその役割、含まれるクラスを表にする
  - コーディング規約を表にする(言語、フレームワークに応じたコーディング規約を定める)
- テスト設計
  - テストの種類と内容を表にする
  - 各テストの目的と方法を説明
  - 実装すべきテストケースを全て表にする
    - 全てのモジュールについて単体テスト
    - 全ての要件定義の機能について結合テスト
    - システム全体の動作を確認するための総合テスト(webやCUIがある場合はユーザー視点でのテストも絶対に定義すること)
- 起動・運用
  - システムは基本的にdocker composeで起動するものとします
  - DBスキーマ、初期ユーザーなどの構築が必要で、システムの初期化が必要なら、システムの起動時に自動的に行うように設計してください
　- 起動方法と操作説明をREADME.mdに記述する旨を設計書に書いておく
